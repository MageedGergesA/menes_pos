# Wave 1 — "Can we legally sell it?" build backlog

The blockers that gate the first paying Egyptian F&B branch. Nothing ships to a
real merchant until W1 is done. This commit **scaffolds the data + endpoint
structure**; the external-authority seams (ETA/ZATCA submit, PSP charge) are
marked `TODO` and return honest "pending / not cleared" responses so the UI can
never print a fake cleared invoice or complete a fake card sale.

### 3. Security — Critical
- **Kill `test123`** ✅: a `post_init_hook` mints a strong `secrets.token_urlsafe(32)`
  on install (replacing unset/`test123`); the token is gone from the frontend
  source; the `/mezze/pos` launcher is now `auth='user'` (staff must log into
  Odoo) and hands the real token to the front-end via a same-origin,
  path-scoped, `SameSite=Strict` cookie, redirecting to a CLEAN url (no
  `?token=`) — so it's never public, never guessable, and no longer sits in the
  URL bar / history / Referer (the token-in-URL residual is closed). The direct
  `?token=` path remains only as the offline-terminal fallback. NEXT (W2): move
  to per-terminal tokens (`mezze.terminal.token`) or full session-auth on the API
  to retire the shared token entirely.
- **Per-user:** `mezze.cashier` PIN login (done). Replace the SUPERUSER scaffold in
  `controllers/sync.py` / `w1.py` `_env()` with the bridge's `_api_env()` (real POS
  user) before production.
- **TODO:** secrets store, HTTPS-only, rate-limit auth endpoints.
