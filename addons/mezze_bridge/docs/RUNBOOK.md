# Mezze Bridge — Runbook (start & test)

Operational guide for running `mezze_bridge` on a dev/test box and exercising it
end-to-end. For the feature inventory see `FEATURE_MATRIX.md`; for the launch
checklist see `GO_LIVE.md`.

---

## 1. Architecture in one screen

`mezze_bridge` is a **thin bridge addon** on stock Odoo 19 Community. A custom
HTML/JS frontend speaks *only* a versioned JSON API (`/mezze/api/v1/`); every
write lands in **native** `pos.order` via `pos.order.sync_from_ui`, so stock,
accounting, loyalty, and MRP food-cost run unmodified. There is no custom order
model.

```
 Frontend SPAs (static/)        HTTP bridge (controllers/)         Native Odoo
 ┌────────────────────┐  JSON   ┌───────────────────────────┐      ┌─────────────────┐
 │ pos.html           │────────▶│ main.py     ~68 endpoints  │─────▶│ pos.order       │
 │ shop.html          │  token  │ w1.py       payments/eta   │      │ stock.quant     │
 │ qr.html            │◀────────│ sync.py     offline engine │      │ account (GL)    │
 │ cfd.html           │  bus    │ hardware.py aggregator.py  │      │ loyalty.card    │
 │ feedback.html      │         │ approval.py (HMAC)         │      │ mrp.bom · bus   │
 └────────────────────┘         └───────────────────────────┘      └─────────────────┘
```

**Cross-cutting mechanics**

- **Transport:** Odoo 19 `type='json2'` bare-REST controllers, `auth='none'` +
  token. The token is accepted from **either** the `X-Mezze-Token` header (curl)
  **or** a `token` field in the JSON body / query string (browser — the framework
  hard-codes the CORS preflight `Allow-Headers` and won't pass a custom header).
- **Per-branch state without new tables:** `ir.config_parameter` JSON keyed by
  `pos.config` id — `store_token_<cfg>`, `eightysix_<cfg>`, `quickkeys_<cfg>`,
  `cfd_<cfg>`.
- **Real-time:** native `bus.bus` channels (`mezze_kds_<cfg>`,
  `mezze_waiter_<cfg>`, `mezze_cfd_update`, …) + a `/bus/poll` fallback.
- **Idempotency:** `pos.order.uuid` (sales), `fire_uuid` (KDS), `last_acked_seq`
  + `mezze.sync.applied` ledger (offline sync).

Depends on: `point_of_sale`, `pos_restaurant`, `stock`, `account`, `bus`, `mrp`,
`loyalty`, `payment_paymob`.

## 2. Start it

### 2.1 Config file

The server runs from an Odoo config that points `addons_path` at the core tree
**plus** the Mezze addons dir, and names the DB. Example (`mezze_test.conf`):

```ini
[options]
addons_path = /home/mageed/odoo_work_19/odoo/odoo/addons,/home/mageed/odoo_work_19/odoo/addons,/home/mageed/odoo_work_19/mezze/addons
db_user = odoo
db_password = odoo
db_name = mezze_test
admin_passwd = admin
log_level = warn
```

> `db_host = False` in the conf means the DB is reached over the local socket
> with the credentials above. A bare `odoo-bin shell -d mezze_test` (without
> `-c <conf>`) then fails with *"no password supplied"* — always pass `-c`.

### 2.2 Run

From the Odoo root (`/home/mageed/odoo_work_19/odoo`):

```bash
CONF=/path/to/mezze_test.conf
python odoo-bin -c "$CONF" -d mezze_test --http-port=8072 -u mezze_bridge
```

### 2.3 Fresh install on a clean DB

```bash
python odoo-bin -c "$CONF" -d NEW_DB \
  -i point_of_sale,pos_restaurant,mrp,loyalty,mezze_bridge --stop-after-init
# then set the token (§3.1) and start normally
```

### 2.4 Restart / rebuild rules

| You changed…                                        | What to run                          |
|-----------------------------------------------------|--------------------------------------|
| `static/*.html` (frontends)                         | **nothing** — reload the browser (cache-busted by file mtime) |
| `controllers/*.py`                                  | **restart** the server (controllers don't hot-reload) |
| `models/*.py`, `security/*.csv`, `__manifest__.py`  | restart **with `-u mezze_bridge`** (schema/ACL upgrade) |

## 3. Credentials & live values

Nothing here is hardcoded in the repo — the token is a real secret stored in the
DB. Discover the live values from the shell:

```bash
python3 odoo-bin shell -c "$CONF" -d mezze_test --no-http <<'PY'
ICP = env['ir.config_parameter'].sudo()
print("TOKEN       =", ICP.get_param('mezze_bridge.api_token'))
print("API_USER_ID =", ICP.get_param('mezze_bridge.api_user_id') or '(default admin)')
cfg = env['pos.config'].search([], limit=1)
print("POS_CONFIG  =", cfg.id, cfg.name)
sess = env['pos.session'].search([('config_id','=',cfg.id),('state','!=','closed')], limit=1)
print("OPEN_SESSION=", sess.id or None, sess.state if sess else '-')
print("STORE_TOKEN =", ICP.get_param('mezze_bridge.store_token_%s' % cfg.id) or '(unset — call /shop/link)')
PY
```

### 3.1 Set / rotate the staff API token

Generated by the `post_init_generate_token` hook on install. To set it explicitly:

```bash
python3 odoo-bin shell -c "$CONF" -d mezze_test --no-http <<'PY'
env['ir.config_parameter'].sudo().set_param('mezze_bridge.api_token', 'CHANGE_ME_secret')
# optional: pin the internal user the API acts as (defaults to admin)
# env['ir.config_parameter'].sudo().set_param('mezze_bridge.api_user_id', '2')
env.cr.commit()
PY
```

## 4. Test it

Set shell vars from the discovery output in §3:

```bash
BASE=http://localhost:8072
TOKEN=<staff api token>
STORE=<store token>
```

### 4.1 curl smoke test (staff API)

```bash
# 1. Health (no auth) — connectivity
curl -s $BASE/mezze/api/v1/health
# {"ok": true, "odoo": "19.0", "module": "mezze_bridge"}

# 2. Bootstrap — catalog + open session; note a product id + payment method id
curl -s -X POST $BASE/mezze/api/v1/bootstrap \
  -H "Content-Type: application/json" -H "X-Mezze-Token: $TOKEN" -d '{}' | head -c 800

# 3. Sync a paid order (idempotent by uuid — rerun returns "duplicate": true)
curl -s -X POST $BASE/mezze/api/v1/orders/sync \
  -H "Content-Type: application/json" -H "X-Mezze-Token: $TOKEN" \
  -d '{"uuid":"smoke-001","session_id":<SID>,
       "lines":[{"product_id":<PID>,"qty":2}],
       "payments":[{"payment_method_id":<PMID>,"amount":0}]}'
```

Fill `<SID>` (open session), `<PID>`, `<PMID>` from the bootstrap response.
Payments' `amount` may be `0` — the server recomputes the tax-inclusive total and
the last tender absorbs the difference, so it always reconciles.

### 4.2 In-browser (the real UI)

```
POS terminal :  $BASE/mezze_bridge/static/pos.html?token=<TOKEN>
Storefront   :  $BASE/mezze_bridge/static/shop.html?store=<STORE>
QR order     :  $BASE/mezze_bridge/static/qr.html?table=<id>&qr=<table token>
CFD (2nd scr):  $BASE/mezze_bridge/static/cfd.html?config=<cfg>

```

Manager panels live under the **Manager** rail button (shift / sync / hardware /
waste / time-clock / feedback / marketing).

### 4.3 Verify in the DB (proof, not vibes)

```bash
python3 odoo-bin shell -c "$CONF" -d mezze_test --no-http <<'PY'
o = env['pos.order'].search([('uuid','=','smoke-001')])
print('order', o.pos_reference, o.state, o.amount_total, 'paid', o.amount_paid)
print('kds tickets', env['mezze.kds.ticket'].search_count([('pos_order_id','=',o.id)]))
PY
```

```bash
python3 odoo-bin shell -c "$CONF" -d mezze_test --no-http <<'PY'
env['mezze.campaign'].sudo().search([]).unlink()
env['sms.sms'].sudo().search([]).unlink()
# env['pos.order'].search([('uuid','like','smoke-%')]).unlink()  # if it's still draft

PY
```

## 5. Troubleshooting

| Symptom | Cause / fix |
|---|---|
| `no password supplied` from `odoo-bin shell` | You omitted `-c <conf>`; the socket needs the conf's `db_user`/`db_password`. |
| `401 unauthorized` on an API call | Token mismatch — re-read `mezze_bridge.api_token` (§3) and pass it in the header or body. |
| `503 server_token_unset` | Token param not set — run §3.1. |
| New model/field/ACL not found after restart | You restarted without `-u mezze_bridge`; models need the upgrade flag. |
| Frontend change not showing | Hard-reload; the page is cache-busted by file mtime, an intermediary cache may be stale. |
| `can't modify Payment Methods while a session is open` | Gift Card PM links to the config only when no session is open; close sessions or link once via setup (see `PAYMOB.md`). |
| QR pay books under a wrong method / "No payment method configured" | Provision a **QR Pay** `pos.payment.method` and add it to the branch's `pos.config` (sessions closed), same as the Gift Card method. Without it, `/qr/pay` falls back to the config's electronic (non-cash, non-tender) method. |
| Endpoints reachable but orders don't post to GL | Accounting posts on **session close**, not per order — close the session to see journal entries. |
